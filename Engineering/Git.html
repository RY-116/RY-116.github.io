# Git 学习笔记

# 1. git commit

Git 仓库中的提交记录保存的是目录下所有文件的快照，就像是把整个目录复制，然后再粘贴一样，但比复制粘贴优雅许多！

Git 希望提交记录尽可能地轻量，因此在每次进行提交时，它并不会盲目地复制整个目录。条件允许的情况下，它会将当前版本与仓库中的上一个版本进行对比，并把所有的差异打包到一起作为一个提交记录。

# 2. git branch

Git 的分支也非常轻量。它们只是简单地指向某个提交纪录 —— 仅此而已。所以许多 Git 爱好者传颂：

早建分支！多用分支！

这是因为即使创建再多的分支也不会造成储存或内存上的开销，并且按逻辑分解工作到不同的分支要比维护那些特别臃肿的分支简单多了。

在将分支和提交记录结合起来后，我们会看到两者如何协作。现在只要记住使用分支其实就相当于在说：“我想基于这个提交以及它所有的父提交进行新的工作。”

main分支的右上角有一个*，这代表着我们当前操控的分支是main分支而不是我们新创立的ry分支

如果此时进行一次 git commit 提交，那么会变成这样

可见我们此次的提交是在main分支上进行的

现在我们去操控新创立的ry分支

执行 git checkout ry

如图，我们已经转换到ry分支上了

注：如果想要在创建一个新分支的同时切换到新分支，可以使用 git checkout -b <newBranchName>

# 3. git merge

在 Git 中合并两个分支时会产生一个特殊的提交记录，它有两个父节点。翻译成自然语言相当于：“我要把这两个父节点本身及它们所有的祖先都包含进来。”

我们首先准备两个经过了不同修改的分支

如图，现在我们在main分支上，而main分支和ry分支上分别有不同的修改

我们使用 git merge ry 将ry分支合并到当前分支————即main分支上，这意味着我们将ry分支上的修改运用到了main分支上

我们可以再切换到ry分支，然后执行 git merge main 来将main分支合并到ry分支上

# 4. git rebase

Rebase 实际上就是取出一系列的提交记录，“复制”它们，然后在另外一个地方逐个的放下去。

Rebase 的优势是可以创造更线性的提交历史

初始情况如下，我们位于ry分支

执行 git rebase main，会在main分支上执行ry分支上执行过的修改

此时再执行 git checkout main 切换到main分支，然后执行 git rebase ry 即可将main分支更新到最新状态

# 5. 分离 HEAD

HEAD通常情况下指向当前分支上最近一次提交记录的分支名

此时HEAD指向main，而main指向c1

我们执行git checkout c1

此时HEAD和main都指向c1

由此可见，通过使用git checkout指令，我们可以很容易地在项目的提交树上移动HEAD

# 6. 相对引用

通过指定提交记录哈希值的方式在 Git 中移动不太方便。在实际应用时，并没有这么漂亮的可视化提交树供我们参考，所以我们就不得不用 git log 来查查看提交记录的哈希值。

并且哈希值在真实的 Git 世界中也会更长（基于 SHA-1，共 40 位）。例如某个提交记录的哈希值可能是 fed2da64c0efc5293610bdd892f82a58e8cbc5d8。

比较令人欣慰的是，Git 对哈希的处理很智能。我们只需要提供能够唯一标识提交记录的前几个字符即可。因此我可以仅输入fed2 而不是上面的一长串字符。

虽然但是，通过哈希值指定提交记录还是很不方便，所以 Git 引入了相对引用

使用 ^ 可以向上移动 1 个提交记录

使用 ~<num> 向上移动多个提交记录，如 ~3

比如 main^ 相当于 main的父提交

执行 git checkout main^

执行 git checkout main^^ 或者 git checkout main~2
